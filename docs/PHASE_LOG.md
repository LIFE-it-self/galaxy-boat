The loss path sets a `stalled` flag, dims the screen with a `0x000000, 0.4` overlay at depth 500, pauses the wake timer via `.paused = true`, and schedules `this.lose()` via a 1s `delayedCall` with a state guard — without the 1s pause, BaseMinigame's LOSE overlay would fire instantly and the stall animation would be invisible.

Win check runs before lose check so if both fire on the same frame, the player earned the win. Splashes pick a random mermaid; left mermaid nudges -15, right mermaid nudges +15, which gives the player a visual hint of which direction the nudge came from (alternative was random sign regardless of which mermaid, but the directional mapping feels fairer).

Notes:
- **PowerMeter required ZERO modifications.** The existing vertical-orientation path (PowerMeter.js:86-91) was written in Session 4 but never exercised until now. MotorboatGame's `new PowerMeter(this, 236, 60, 12, 120, { orientation: 'vertical', ... })` call just worked — the fill rectangle correctly shrinks from bottom-up and the color tiers (green > 50%, yellow 25-50%, red < 25%) still apply.
- **Tweened circles over ParticleEmitter**: zero particle code exists in the codebase, and Phaser 3.80's ParticleEmitter API has enough footguns (texture keys, emit configs, dynamic rate updates) that a session whose whole purpose is playability is not the right place to debut it. PipeSmoke's existing smoke-ring tween pattern (`this.add.circle` + `this.tweens.add({ targets, alpha: 0, y: ..., onComplete: destroy })`) is the reference. Both MotorboatGame wake spray and MermaidShower droplets/splashes follow it. Peak concurrent live tweens are ~30-50 during high-power motorboat play — Phaser handles this cheaply and each tween self-destructs via `onComplete`.
- **`stalled` flag rationale.** BaseMinigame.lose() immediately shows the LOSE overlay, which blocks all further rendering of the underlying scene. If MotorboatGame called `this.lose()` the moment the PowerMeter hit 0, the "boat stalls visually" effect (darken, freeze mermaids, stop wake) would never be seen. The `stalled` flag lets update() suppress new wake spawns, pause the wake timer, freeze mermaid bounce math, and add a dim overlay at depth 500 — all for 1 second before `this.lose()` fires via delayedCall. The inner callback guards on `this.state === 'PLAY'` (the Session 4 pattern) in case something else ended the scene during the delay.
- **Held-key polling vs keydown events.** Session 6 troubleshooting explicitly calls this out: MermaidShower's LEFT/RIGHT input uses `this.input.keyboard.isDown(...)` (via `cursors.left.isDown`) in update() because `keydown` events fire once on press, not continuously. With polled input, `temperature ±= 25 * delta / 1000` integrates smoothly over held-key duration.
- **In-zone accumulator pause semantics.** The spec is explicit: "If currently in green, the counter ticks up. If not, it doesn't decrease, just pauses." This means the player can yo-yo in and out of the green zone and progressively accumulate 10 seconds without being punished for leaving. Implementation: `if (temperature >= greenMin && temperature <= greenMax) this.inZoneMs += delta;` — no else branch, no decay.
- **Win-before-lose ordering** in MermaidShower `update()`. Both win and lose branches return early on fire, so no double-fire is possible, but check order matters for the one-frame edge case where the player crosses 10s in-zone on the same frame the total timer expires. Win takes priority.
- **Sequence guard test (planned).** Fresh game → walk to Bridge → step on (12, 4) = mermaid-shower → `SequenceGuard.assertCanStartRitual(game, 3)` is called from `OverworldScene.startMinigameForLevel` BEFORE MermaidShower ever boots. `ritualProgress.length + 1 = 1`, `stepNumber = 3` → mismatch → `'hurricane-fail'` emitted → HUDScene's listener handles the banner. Pipe-only → same result (expected 2, attempted 3). Pipe + Dinner → `ritualProgress = [1, 2]`, expected 3, attempted 3 → pass → MermaidShower launches normally. Final playtest happens in a real browser (headless preview is too slow to walk through the full Act 1 → Act 2 → Act 3 flow by hand).
- **`PowerMeter.value` is still a plain property** (Session 4 lesson). MotorboatGame reads it via `this.powerMeter.value` and never assigns. All bumps go through `add()`, all drains through `decay()`.
- **Shutdown hygiene.** Both new minigames register `this.events.once('shutdown', this.shutdownGame, this)` and explicitly remove their timers (`.remove(false)`) and destroy their PowerMeter in that handler. PipeSmoke relied on Phaser's auto-cleanup and this was fine, but the timer removal is worth being explicit about because `this.time.addEvent({ loop: true })` timers hold scene refs and can get loud on shutdown if anything tries to fire after the scene is torn down.

### Session 5
Act 2 playable end-to-end. New files:
- `src/scenes/minigames/ScubaDiveGame.js` — top-down underwater collection minigame. First scene in the project to use **Arcade Physics**. Off-screen fish are destroyed in `update()` using `group.getChildren().slice().forEach(...)` — without the `.slice()` copy, destroying during iteration would skip siblings. Win at `hits >= targetFish` (default 10), lose at `lives <= 0` (default 3) or at the 30s timer expiry. Touch d-pad inlined (doesn't share OverworldScene's buttons because the minigame scene is parallel-independent). `PLAYER_SPEED = 60 px/sec`.

Notes:
- The first `RhythmBar` note arrives at `tMs = 1000` with `NOTE_TRAVEL_MS = 1500`, so its spawn offset is `-500`. The partial-travel branch handles this by computing `fraction = 500 / 1500 = 1/3`, spawning at `rightEdgeX - (1/3) * barSpan`, and tweening the remaining 1000ms to the hit zone. Without this, the first note would either be missing or misaligned.
- All `delayedCall` callbacks in the new minigames check `this.state === 'PLAY'` before touching anything. This is essential because once `win()` or `lose()` fires, the state flips away from `'PLAY'` and the result overlay runs for ~1.2s before the scene actually shuts down — any in-flight callbacks must no-op during that window or they'll touch destroyed objects and crash.

- `BaseMinigame.lose()` does NOT call `markMinigameComplete`, so a failed `coke-drink` attempt leaves the trigger active and the player can re-step on it to retry. This is what makes the 5-fail hurricane test work naturally — repeated CokeDrink losses tick the failure counter without unblocking the trigger.

- `src/scenes/DialogScene.js` — bottom-of-screen typewriter dialog box (30ms/char). First press fast-forwards line, second advances. Emits `'dialog-complete'` on EventBus when done. 50ms input grace before listeners attach to swallow the launching pointerdown.
- `src/scenes/TransitionScene.js` — black title card with instruction + location. 2s auto-advance OR space/tap to skip. `advanced` flag prevents double-fire.
- `src/scenes/minigames/BaseMinigame.js` — **sacred file, do not edit from minigame subclasses.** Named export. Exposes `setupGame()`/`evaluate()` overrides and `win()`/`lose()` helpers that handle state, GameStateManager bookkeeping, the result overlay, and the return-scene start.
- `src/scenes/minigames/PlaceholderGame.js` — first BaseMinigame subclass. SPACE wins, X loses, top-half tap wins, bottom-half tap loses.
- `src/objects/Cody.js` — green 16×16 NPC with yellow `!` marker. `isAdjacentTo()` uses Manhattan distance.
