Dark blue bedroom, gray bed, Cody (green 16×16) on the bed, two pink mermaids at (96, 145)/(160, 145) with a yoyo `y: '-=4'` bounce, "TAP LULLABY" label, a `hits/beats` counter, and a `RhythmBar` at `(16, 188, 224, 20)` — same coordinates as `CokeDrinkGame`. Notes scheduled `addNote(1000 + i * noteSpacingMs)` for `i = 0..beats-1` and evaluation fires at `lastNoteAt + 300` via a single state-guarded `delayedCall`. Both `pointerdown` and `keydown-SPACE` call `this.tryHit()` which defers to `rhythmBar.hit()`. Does NOT call `assertCanStartRitual` (non-ritual level — that guard runs externally in `OverworldScene.startMinigameForLevel` for ritual steps only).
- `src/scenes/minigames/MermaidNap.js` — **Ritual Step 4, the final step.** Four noises fire at deterministic module-level times (`NOISE_TIMES = [3000, 7500, 12000, 16500]`) with module-level labels (`NOISE_LABELS = ['SEAGULL!', 'FOGHORN!', 'WAVE CRASH!', 'PARROT!']`). Each noise opens a 2000ms shush window. `tryShush()` during the window resolves the noise; missing it calls `this.meter.add(-wakeOnMiss)`. The meter is a `PowerMeter` at `(16, 188, 224, 12)` with **inverted semantics** — initial value 100 (not 0), `decayPerSec: 0`, label "CODY ASLEEP". Meter value represents sleep depth: 100 = fully asleep (green), 0 = fully awake (lose). Lose triggers when `meter.value <= 0`; win triggers when `elapsedMs >= totalDurationMs` and `meter.value > 0`. The win-before-lose check order in `update()` matches MermaidShower's pattern. BaseMinigame's `win()` auto-records ritual step 4 because `isRitual: true` in the level config — which makes `ritualProgress.length` hit 4 and `markRitualStep` emits `'victory'` on the EventBus. CutsceneRouter catches it. All noise timers are tracked in `this.noiseTimers[]` and removed in `shutdownGame()` alongside `this.shushTimer`.
- `src/scenes/CutsceneScene.js` — Data-driven cutscene player. `init(data)` reads `mode` (`'fail'` | `'victory'` | anything else → bounces to menu). `create()` dispatches and installs a single `shutdown` listener. Tracks every `delayedCall` handle in `this.timers[]` via `pushTimer()` and iterates them in `shutdown()` calling `.remove(false)` — essential because `returnToMenu()` can fire early (RETRY clicked before Aquaman shows) and any in-flight callbacks would touch a dead scene. `returning` flag guards `returnToMenu()` against double-entry (pointer + SPACE + ENTER all wired). **Fail mode** sequence: 1000ms camera shake (`0.01` amplitude) + green Cody spinning out (`angle: 720, scale: 6, duration: 1500, ease: 'Quad.In'`) → `showHurricane()` at 1500ms (deep blue-gray background, rotating gray circle at depth 0, "HURRICANE"/"FLORIDA SINKS" text at depth 100) → `showAquaman()` at 4000ms (black cover at **depth 500** to hide the hurricane layer, teal throne at 501, Cody on throne at 502, yellow trident rect at 502, "CODY RULES THE DEEP" at 503) → `showRetryButton()` at 7000ms (RETRY rectangle at depth 510 with white stroke, text at 511, plus wired SPACE/ENTER keys). Keyboard binding is deferred to `showRetryButton` specifically so the player can't skip the entire sequence with a fresh SPACE press. **Victory mode** fades in over 1000ms onto a beach scene: sky `rect(128, 60, 256, 120, 0x60a0ff)` (y=0..120) + sun circle + beach `rect(128, 172, 256, 104, 0xf0d080)` (y=120..224) + Cody on the sand. Credits (13 lines, including the baked-in "CODY GOT OFF THE BOAT." title) scroll from y=240 to y=-200 over 12000ms linear, then `returnToMenu()` fires. A 400ms input grace precedes attaching the skip handler so the pointerdown that just won MermaidNap doesn't immediately dismiss the cutscene. `returnToMenu()` calls `GameStateManager.reset(this.game)` (zeros `failureCount`, `ritualProgress`, `completedMinigames`, `talkedToCody`, `selectedCodyVariant`, `currentAct`), then iterates `this.scene.manager.scenes` and stops anything active/paused that isn't CutsceneScene or MainMenuScene, then `scene.start('MainMenuScene')`.
- `src/systems/CutsceneRouter.js` — Module-scope event bridge. Exports `registerCutsceneRouter(game)` which is called ONCE from `index.js` after `new Phaser.Game(config)`. Registers two EventBus listeners (`'hurricane-fail'` → `route(game, 'fail', data)`, `'victory'` → `route(game, 'victory', data)`). `route()` uses two guards: a `routing` flag (for same-frame double-emits) and an `isActive('CutsceneScene')` check (for already-running cutscenes). Critically, the scene handoff is deferred with `Promise.resolve().then(...)` — `BaseMinigame.win() → markRitualStep() → EventBus.emit('victory')` is a synchronous call chain, and stopping scenes mid-update historically caused "Cannot read property 'sys' of undefined" crashes. The microtask push defers the mutation to the next event-loop tick, after Phaser's current update call returns. Inside the microtask, the stop-loop runs BEFORE `scene.start('CutsceneScene')` — otherwise CutsceneScene would get added to the active list and the loop would stop it immediately. A module-level `routerRegistered` flag makes the `register` function idempotent, which matters for Vite HMR (the module re-executes on save but the flag blocks re-binding the listeners).

- **The game is playable end-to-end.** Three endings verified via the preview eval tool (headless browser is too slow to wall-clock a 15-minute playthrough, so the endings were triggered by direct `EventBus.emit()` calls once every code path was in place):

This confirms that registering listeners at module scope (via `FailHandler.js` side-effect import + `registerCutsceneRouter` idempotent call) survives `scene.start/stop` without accumulating.
- **MermaidNap PowerMeter inversion rationale.** PowerMeter's internal color tiers assume "bigger = better" (green at >50%, yellow 25-50%, red <25%). Wakefulness naturally grows upward, but if we tracked it that way the bar would be green while Cody was asleep and then turn red as he woke up — which is the *opposite* of what players expect ("the fill bar is draining, I'm losing"). Inverting (track sleep level from 100 downward) means the bar visually drains away from the good state, which reads correctly without any PowerMeter-level modification. This is documented in CLAUDE.md's UI components section.
- **Two CutsceneScene polish bugs caught during playtest.**
  1. **Depth bleed in `showAquaman`**. Initial implementation used a depth-0 black cover rect to hide the hurricane backdrop from `showHurricane`. That backdrop's HURRICANE/FLORIDA SINKS text had been created with `setDepth(100)`, so the cover drew *under* them and the hurricane labels bled through on top of the throne. Fixed by giving every element in showAquaman an explicit depth: cover at 500, throne at 501, Cody on throne at 502, trident at 502, result text lines at 503. Then `showRetryButton` uses 510/511 to sit cleanly on top. Screenshot diff confirmed the fix.
  2. **Sky/beach gap in `playVictory`**. The plan's original numbers (sky `rect(128, 56, 256, 112)` covering y=0..112, beach `rect(128, 180, 256, 88)` covering y=136..224) left a 24px black horizontal band in the middle of the beach scene. Same edit moved Cody-on-sand from y=192 to y=160 so he wasn't clipping the bottom edge. - **Microtask deferral in CutsceneRouter is load-bearing.** Without it, `BaseMinigame.win() → markRitualStep(4) → EventBus.emit('victory') → CutsceneRouter.route() → stopAllExcept → scene.stop('MermaidNap')` runs synchronously inside MermaidNap's own `update()` or timer callback. Stopping the scene that is currently mid-update has historically caused "Cannot read property 'sys' of undefined" crashes deep in Phaser's scene manager. The `Promise.resolve().then(...)` kicks the stop-and-start to the next microtask, after the current frame's update completes. Alternative designs considered: (a) `game.events.once('postrender', fn)` — but router doesn't have a game reference at that layer and timing is fuzzy; (b) `delayedCall(0, fn)` — works but adds a full frame of latency and leaks a timer handle; (c) current approach — zero frame latency, no handle to track, standard ES semantics.
- **Double-cutscene prevention.** If ritual step 4 wins AND failure 5 also fires on the same frame (theoretically possible if a lose path somehow runs during a win overlay), the `routing` flag + `isActive('CutsceneScene')` check in `route()` ensures only the first cutscene starts. The second call becomes a no-op.
- **Victory race with BaseMinigame's result overlay.** `BaseMinigame.win()` calls `markRitualStep(4)` BEFORE its 1200ms `showResultOverlay` delay. This is the desired behavior (no stale result card lingering on top of the cutscene) and is explicitly preserved.
- **Sequence guard is step-4 aware automatically.** The guard checks `ritualProgress.length + 1 === attemptedStep`, so if a player walks into cabin-corridor and tries MermaidNap (step 4) without having done MermaidShower (step 3) first, they get an immediate `'hurricane-fail'` with `expected: 3, attempted: 4`. No code change was needed in SequenceGuard itself for Session 7.
- **Cabin Corridor door-pair double-check.** The Bar↔Galley and Main Deck↔Bridge existing pairs both use floor-tile spawn targets (not door-tile spawns) to prevent the door tile from firing a second transition on the same frame the player arrives. Cabin Corridor follows the same rule: main-deck door at col 15 → spawn at cabin-corridor col 1 (floor); cabin-corridor door at col 0 → spawn at main-deck col 14 (floor). Walked through both directions; no bounce loops.
- **Two tests found no bugs and were kept as verification-only**: (1) the RETRY button's pointer + SPACE + ENTER fire paths all route through `returnToMenu()` which is guarded by the `returning` flag, so triple-firing is a no-op on calls 2 and 3; (2) stopping every scene except CutsceneScene doesn't race with MainMenuScene because `returnToMenu` first stops all gameplay/minigame/HUD scenes, then calls `scene.start('MainMenuScene')` which implicitly stops CutsceneScene itself. Vertical `PowerMeter` on the right (first use of the vertical orientation in the codebase) at x=236, y=60..180, max 100, initial 60, `decayPerSec: 50`.
